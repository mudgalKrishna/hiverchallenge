# CloudSync Suggested-Response Pipeline

This repository contains an end-to-end Retrieval-Augmented Generation (RAG) pipeline for automatically generating and evaluating customer support email replies for "CloudSync".

## Project Overview & Deep Data Approach

### The Challenge of LLM-Generated Data
We aggregated synthetic customer support datasets generated by three different frontier LLMs: Copilot, Gemini, and Claude. When working with raw LLM-generated datasets at scale, data formatting is a massive hurdle. The raw data (especially the Gemini dataset) contained highly malformed JSON fragments, missing commas, trailing brackets, and stray conversational text.

### The Solution: Indestructible Streaming Parser
Instead of manually cleaning thousands of lines of broken JSON, we built a highly robust **streaming data loader (`data_loader.py`)**. 
* It reads files character-by-character using a custom fallback decoder.
* It automatically skips stray words and unparsable characters without crashing the pipeline.
* It validates every single record against a strict schema and silently drops fragmented or incomplete records, guaranteeing 100% data integrity for the downstream RAG pipeline.

### Dataset Schema & Fields
Every valid record in our Knowledge Base is parsed into a strict structure containing the following required fields:
* `id`: A unique identifier for the conversation.
* `intent`: A short categorical label (e.g., "billing_issue", "mfa_setup"). This allows us to stratify our train/test splits so the system is evaluated on a balanced distribution of customer problems.
* `incoming_email`: The raw text of the customer's support request. Our `sentence-transformers` model embeds this exact text to calculate cosine similarity during retrieval.
* `ground_truth_reply`: The "gold standard" reply written by an expert agent. This serves as the few-shot style and content reference for the generator model.
* `gold_requirements`: A list of specific steps or actions the reply *must* accomplish to successfully solve the user's issue.
* `must_include`: Mandatory phrases, links, or facts that must be present.
* `must_not_include`: Restricted phrases (e.g., promising a refund when company policy forbids it).

### Why this specific structure?
This rich structure serves a powerful dual purpose:
1. **For Generation (RAG):** The `incoming_email` and `ground_truth_reply` are used by the generator to build highly relevant, dynamic few-shot prompts. The LLM learns the tone, formatting, and standard operating procedures directly from these retrieved examples.
2. **For Evaluation (LLM-as-a-Judge):** The `gold_requirements`, `must_include`, and `must_not_include` fields are explicitly passed to our automated Judge. Instead of relying on fragile keyword-matching to grade the AI, the Judge uses these exact constraints to rigorously score the AI's generated response on **Intent Fulfillment** and **Constraint Adherence** on a 1-5 scale.

## ⚠️ Why Open-Source Inference APIs? (Project Limitations)

**Due to time constraints and a lack of compute hardware/budget as a student, it was not feasible to locally fine-tune a custom LLM.** 

Instead, this pipeline relies on zero-shot/few-shot prompting via external APIs. We heavily leveraged free-tier API providers and open-source models to prove the architecture. Because we relied on free-tier endpoints, you may notice occasional instability issues in the logs:
* **Empty Reasoning Traces:** Large free "thinking" models sometimes exhaust their token limits on internal thoughts, returning blank content. Our `llm_provider.py` implements a robust fallback to catch this.
* **429 Rate Limits:** Free nodes strictly limit requests.

**To achieve production-level stability:** We highly recommend using a paid API key or deploying an open-source model directly onto Hugging Face Spaces. The pipeline uses a Strategy Pattern (`llm_provider.py`) making it instantly compatible with multiple providers.

## How to Run

1. **Install Dependencies:**
   ```bash
   pip install -r requirements.txt
   ```

2. **Set up the LLM Provider:**
   The pipeline supports Hugging Face Spaces, OpenRouter, Groq, OpenAI, and Anthropic (via OpenRouter). Choose **ONE** of the configurations below to put in your environment or Colab cell:

   ### Option A: Hugging Face Spaces / Inference API
   ```python
   import os
   os.environ["LLM_PROVIDER"] = "huggingface"
   os.environ["HF_API_TOKEN"] = "hf_YOUR_TOKEN"
   os.environ["HF_MODEL_ID"] = "mistralai/Mistral-7B-Instruct-v0.3"
   # Optional: os.environ["HF_API_URL"] = "https://your-custom-space.hf.space"
   ```

   ### Option B: OpenRouter
   ```python
   import os
   os.environ["LLM_PROVIDER"] = "standard"
   os.environ["STANDARD_API_BASE"] = "https://openrouter.ai/api/v1"
   os.environ["STANDARD_API_KEY"] = "sk-or-v1-YOUR_OPENROUTER_KEY"
   os.environ["STANDARD_MODEL"] = "meta-llama/llama-3.1-8b-instruct"
   ```

   ### Option C: Groq
   ```python
   import os
   os.environ["LLM_PROVIDER"] = "standard"
   os.environ["STANDARD_API_BASE"] = "https://api.groq.com/openai/v1"
   os.environ["STANDARD_API_KEY"] = "gsk_YOUR_GROQ_KEY"
   os.environ["STANDARD_MODEL"] = "llama-3.3-70b-versatile"
   ```

   ### Option D: OpenAI
   ```python
   import os
   os.environ["LLM_PROVIDER"] = "standard"
   os.environ["STANDARD_API_KEY"] = "sk-YOUR_OPENAI_KEY"
   os.environ["STANDARD_MODEL"] = "gpt-4o-mini"
   ```

   ### Option E: Anthropic (Claude via OpenRouter)
   *Note: Because our code uses the standard OpenAI-compatible format, the easiest way to access Anthropic models without rewriting the API wrapper is to route the request through OpenRouter.*
   ```python
   import os
   os.environ["LLM_PROVIDER"] = "standard"
   os.environ["STANDARD_API_BASE"] = "https://openrouter.ai/api/v1"
   os.environ["STANDARD_API_KEY"] = "sk-or-v1-YOUR_OPENROUTER_KEY"
   os.environ["STANDARD_MODEL"] = "anthropic/claude-3.5-sonnet"
   ```

3. **Run Pipeline:**
   ```bash
   python main.py
   ```

The script will output a clean `results.csv` containing the `incoming_email`, the `generated_reply`, and the `overall_score` (1-5).
